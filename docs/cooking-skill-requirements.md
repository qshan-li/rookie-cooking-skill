# 做菜 Skill 需求文档

状态：需求整理版  
来源：Obsidian 项目 `1_Projects/20_做菜skill` 下的 v0.2 设计文档、v0.3 记忆层讨论、新功能取舍讨论  
更新日期：2026-05-26

## 1. 产品定位

做菜 Skill 是一个面向理性新手的可解释家常菜执行系统。

它的目标不是做一个普通菜谱库，也不是实时陪做 App，而是把中餐菜谱中常见的模糊经验转成可以执行、可以打印、可以复盘的厨房文档。

核心定义：

> 把“适量、大火、炒熟、收汁”等模糊经验，转成克数、时间、状态判断、失败诊断、替代方案和原理解释。

最终交付物应优先是一份可以放在厨房台面上、可以离线执行的 Markdown / PDF / 纸质菜谱文档。

## 2. 核心价值

传统菜谱回答“今天这道菜怎么做”。本 Skill 需要进一步回答：

- 为什么这么做。
- 如何稳定复现。
- 做坏了如何判断原因。
- 下次如何调整。
- 如何把单道菜经验迁移到其他菜。
- 如何根据用户的设备、口味和历史反馈生成更适合的厨房文档。

核心价值不是菜谱数量，而是把做菜中的隐性经验显性化：

- 把“适量”转成默认克数和调整规则。
- 把“大火”转成燃气灶、电磁炉和状态判断三套描述。
- 把“炒熟”转成时间区间、状态判断和食品安全标准。
- 把“翻炒片刻”转成操作动作、持续时间和目标状态。
- 把“失败了”转成原因诊断和下次修正方案。
- 把用户稳定偏好、设备条件和菜谱反馈变成下次生成时可用的适配信息。

## 3. 目标用户

核心用户：

- 能理解数字、比例和流程逻辑。
- 不需要食品科学背景，但愿意理解关键原因。
- 愿意为了稳定复现接受稍高的信息密度。
- 本质上是“脑子会了，手还没会”的做饭新手。

典型痛点：

- 看得懂菜谱，但做出来不稳定。
- 不理解“适量”“少许”“大火”“炒熟”。
- 不知道失败原因，只能归因于“没天赋”。
- 不知道设备、锅具、食材状态会怎样影响结果。
- 不知道哪些参数可以改，哪些参数不能乱改。
- 想通过理解原理提升，而不是死记菜谱。

非目标用户：

- 只想看 30 秒短视频做菜的人。
- 完全不愿意称重、计时、理解逻辑的人。
- 已经有稳定手感的熟练厨师。
- 追求高级餐厅级摆盘或复杂料理的人。

## 4. 产品原则

精确，但不伪精确：

- 默认值必须明确。
- 允许区间必须明确。
- 状态判断必须明确。
- 调整规则必须明确。
- 不把不稳定参数强行写成唯一值。

操作优先，原理服务操作：

- 原理解释只服务当前操作、判断、纠错和迁移。
- 默认输出不写成长篇科普。
- 用户追问时再进入展开解释或 Learning 模式。

菜谱与原理分离：

- 菜谱负责执行。
- 原理卡负责解释。
- 目标模式只展示必要解释。
- 复杂解释进入 Learning 模式。

安全优先于口感：

- 涉及肉类、禽类、海鲜、剩菜、解冻、复热时，必须优先给出食品安全底线。
- 当口感建议与安全建议冲突时，安全优先。

厨房场景友好：

- 用户做饭时可能手上有油或水，不方便频繁操作屏幕。
- 必须支持完整解释版和厨房执行版 / 打印版。
- 厨房执行版默认少解释、强步骤、强状态判断。

## 5. 产品边界

明确不做独立实时陪做模式：

- 不把产品设计成手机上的实时下一步助手。
- 不依赖用户做饭时持续对话。
- 不要求用户边做饭边频繁滑动屏幕。

明确不做内置计时器：

- 不做系统级计时提醒。
- 不做跨平台通知。
- 不做自动闹钟。

对应真实需求的转化方式：

- “实时下一步”转成厨房执行版、大字号步骤、明确步骤编号、每步时间、目标状态、失败信号。
- “计时提醒”转成文档中的关键计时点、长时间炖煮的闹钟建议、多菜排程中的检查节点。

## 6. 使用模式

### 6.1 目标模式：我要做一道菜

用户只输入菜名时，Skill 不先追问，直接按默认配置生成菜谱。

默认配置：

- 默认 2 人份。
- 默认有燃气灶或电磁炉。
- 默认有普通炒锅。
- 默认无厨房温度计。
- 默认有手机计时器。
- 默认可使用电子秤，同时提供无秤替代判断。

输出必须包含：

- 完整菜谱。
- 精确用量。
- 时间区间。
- 火力 / 设备描述。
- 每步目标状态。
- 每步必要的“为什么”。
- 常见错误和失败诊断。
- 替代方案。
- 可打印厨房版。

结尾应提示用户可以提供人数、锅具、电子秤、温度计等信息重新适配。

### 6.2 Learning 模式：我要理解一个原理

用户询问原理时，Skill 输出原理卡。

原理卡必须包含：

- 一句话解释。
- 原理解释。
- 实际应用。
- 反直觉案例。
- 常见误区。
- 适用菜谱。
- “用这道菜试试”的反向链接。

Learning 模式也需要读取相关用户记忆。例如用户没有温度计时，解释油温应优先讲筷子冒泡法，而不是只讲温度计法。

### 6.3 快速模式

适合用户只想马上做饭。

特点：

- 不先追问。
- 步骤短。
- “为什么”只给一句话。
- 保留关键状态判断。
- 保留失败诊断。

### 6.4 精准模式

适合用户愿意称重、计时、控温。

输出应包含：

- 精确克数。
- 温度范围。
- 时间范围。
- 食材重量缩放。
- 火力适配。
- 中心温度 / 油温建议。
- 更详细的原理解释。

## 7. 输出形态

必须支持两种输出：

- 完整解释版。
- 厨房执行版 / 打印版。

厨房执行版要求：

- 正文 14pt 以上。
- 步骤编号醒目。
- 高对比度。
- 行距宽。
- 每一步不超过 3 行。
- 默认不打印原理长解释。
- 保留关键安全提示。
- 单道菜尽量控制在 1-2 页 A4。
- 顶部保留食材清单，可作为购物清单使用。

V1 优先支持：

- 生成厨房友好的 Markdown。
- 生成可打印 PDF。
- 用户自行下载或传输到手机 / 打印机。

V1 不优先支持：

- 自动检测系统打印机。
- 自动调用 `lp` / `lpr`。
- 跨平台直接打印。

## 8. 功能全集与优先级

所有功能都服务于同一个核心：生成一份更适合用户、更稳定、更可执行的厨房文档。

### 8.1 P0：先保证菜谱本身靠谱

替代方案引擎：

- 没有某个食材时，给出可替代项和原因。
- 没有某个工具时，给出可执行替代判断。
- 电磁炉、燃气灶、铁锅、不粘锅等设备差异需要影响操作参数。

菜谱验证与评分系统：

- 每道菜谱必须经过执行完整性、参数可靠性、原理匹配、安全检查、厨房友好性检查。
- 不合格菜谱不能作为标杆菜进入知识库。

失败诊断增强：

- 每道菜必须包含通用失败诊断和专属失败诊断。
- 诊断必须给出可能原因和下次调整方案。

厨房执行版强化：

- 把实时陪做需求转成可打印、可扫读的厨房步骤文档。
- 必须标注关键计时点和检查节点。

### 8.2 P1：个性化和一餐规划

用户记忆层：

- 记录稳定的口味、设备、分量、忌口、做菜历史和失败反馈。
- 生成菜谱时自动读取相关记忆并适配。
- 输出时说明使用了哪些偏好。

菜谱个性化调参器：

- 根据人数、口味、设备、锅具、历史失败反馈调整用量和步骤。
- 支持本次覆盖默认值，但不自动污染长期偏好。

购物清单生成：

- 从单道菜或一餐菜单生成食材和调料清单。
- 顶部食材清单应可直接作为简版购物清单使用。

一餐菜单规划：

- 支持从“做一顿饭”角度规划多道菜。
- 需要考虑荤素、主食、口味重复、时间压力和设备限制。

并行时间线 / 厨房排程：

- 多菜时输出先后顺序和并行时间线。
- 标注需要设置闹钟或检查的节点。

菜谱版本管理：

- 菜谱修改后应能区分版本。
- 用户反馈导致的单菜调整应能追溯。

### 8.3 P2：扩展能力

冰箱库存 / 现有食材模式：

- 用户给出现有食材时，基于库存推荐可做菜。
- 当天库存默认只作为本次上下文，不长期记忆，除非用户明确要求。

营养与热量估算：

- 可估算每份热量和宏量营养。
- 只能作为辅助估算，不做医学诊断或治疗建议。

风味偏好校准测试：

- 用少量问题或菜谱反馈校准咸淡、甜酸、辣度、油量偏好。

菜谱来源与可信度标注：

- 引用或改写外部菜谱时保留来源说明。
- 对来源可信度和改写程度做标注。

菜谱导入与改写：

- 可从已有菜谱中导入结构。
- 导入后必须按本 Skill 模板重写、review 和补全状态判断。

家庭成员偏好：

- 后续支持多个家庭成员口味和忌口。
- 不应在基础版本中引入过度复杂的用户模型。

## 9. 菜系与内容范围

V1 以中餐家常菜为主，覆盖炒、炖、蒸、煮、凉拌，目标 20-30 道打底，并包含少量快手西餐作为对照和补充。

中餐家常菜方向：

- 炒菜类。
- 肉丝 / 肉片类。
- 鸡蛋类。
- 蒸菜类。
- 炖煮类。
- 凉拌类。
- 简单汤类。

候选菜谱：

| 类型 | 菜名 | 核心训练点 |
| --- | --- | --- |
| 鸡蛋类 | 番茄炒蛋 | 蛋白质凝固、酸甜咸平衡、出水控制 |
| 鸡蛋类 | 蒸蛋羹 | 蛋水比例、过滤、蒸汽控制、凝固温度 |
| 肉丝类 | 青椒肉丝 | 上浆、滑炒、锅温、肉类嫩化 |
| 肉丝类 | 鱼香肉丝 | 调味平衡、上浆、复合酱汁 |
| 蔬菜类 | 清炒小青菜 | 火力、水分、盐的加入时机 |
| 蔬菜类 | 蒜蓉西兰花 | 焯水、定色、蒜香释放 |
| 炖煮类 | 红烧肉 | 美拉德、焦糖化、胶原蛋白转化 |
| 炖煮类 | 土豆炖牛腩 | 胶原蛋白、淀粉、长时间炖煮 |
| 蒸菜类 | 清蒸鱼 | 蛋白质熟化、去腥、蒸制火候 |
| 汤类 | 紫菜蛋花汤 | 蛋花形成、鲜味、盐度控制 |
| 凉拌类 | 凉拌黄瓜 | 盐腌脱水、酸咸甜平衡 |
| 豆腐类 | 麻婆豆腐 | 勾芡、乳化、油脂承载香味 |

快手西餐方向：

| 菜名 | 核心训练点 |
| --- | --- |
| 番茄肉酱意面 | 淀粉水乳化、酸味平衡、酱汁挂面 |
| 煎牛排 | 表面干燥、美拉德、中心温度、静置 |
| 沙拉油醋汁 | 乳化、酸油比例、调味平衡 |

交付节奏：

- 先做 3-5 道标杆菜，打磨模板和原理链接。
- 再按相同结构扩展到 20-30 道。

首批标杆菜：

- 番茄炒蛋。
- 青椒肉丝。
- 蒸蛋羹。
- 清炒小青菜。
- 红烧肉。

## 10. 精度标准

所有菜谱默认提供：

- 固体用量：g。
- 液体用量：ml。
- 食材数量：个数 + 约重。
- 温度：摄氏度。
- 时间：秒 / 分钟。
- 火力：设备相关描述。

原则上避免单独使用：

- 适量。
- 少许。
- 若干。
- 一勺。
- 一把。
- 炒一会儿。
- 差不多熟了。

如果必须使用感官描述，必须搭配可观察标准。

参数类型：

- 可精确参数：盐、生抽、鸡蛋重量、清水量、固定蒸制时间等。
- 区间参数：炒青菜 45-75 秒，并给出目标状态。
- 自适应参数：咸淡、辣度、酸甜比、汤汁多少、软烂程度、是否低油等。

## 11. 火力与温度标准

中餐火力描述必须转成可操作描述。

| 传统描述 | 燃气灶参考 | 电磁炉参考 | 状态判断 |
| --- | --- | --- | --- |
| 大火 | 外圈火焰接近或刚能包住锅底 | 1800-2200W | 油流动明显，食材下锅有强烈滋啦声 |
| 中火 | 火焰高度约为锅底一半 | 1000-1500W | 食材持续冒泡但不剧烈焦化 |
| 小火 | 火焰集中在锅底中心 | 400-800W | 汤汁轻微冒泡或缓慢收汁 |

火力标准必须与锅具、食材量结合，不作为绝对值。

有温度计时给温度；无温度计时给感官判断。

示例判断：

- 油温 150-160°C：筷子插入油中，周围出现稳定小泡。
- 锅预热：手离锅面 15cm 有明显热感，倒油后油纹快速扩散。
- 鸡肉熟化：切开最厚处无粉红色，肉汁清澈；但需说明感官判断不如温度计可靠。

## 12. 人份缩放规则

中餐菜谱不能简单整体线性缩放。

缩放规则：

| 项目 | 缩放规则 |
| --- | --- |
| 主料 | 通常线性缩放 |
| 盐 | 小份近似线性，4 人份以上略低于线性 |
| 生抽 / 酱油 | 接近线性，但需考虑本身盐度 |
| 糖 | 根据酸度和口味调整，不完全线性 |
| 油 | 不建议严格线性，通常有上限 |
| 水 / 高汤 | 根据锅具蒸发面积和炖煮时间调整 |
| 炒制时间 | 不随人份线性增加，以状态为准 |
| 炖煮时间 | 主要由食材尺寸决定，不由总量决定 |

分批规则：

- 炒菜需要考虑锅容量。
- 当总生重超过约 500g 时，应考虑分批处理。
- 原因是食材过多会迅速拉低锅温，导致出水和口感变差。

## 13. 解释层级

为了避免目标模式变成科普长文，解释分三层：

- Level 1：一句话解释，默认展示。
- Level 2：展开解释，用户追问时展示。
- Level 3：Learning 模式，进入完整原理卡。

每个“为什么”都必须服务当前操作。

## 14. 知识库设计

知识库采用可引用原理卡，不写成长篇文章。

每张原理卡必须包含：

- ID。
- 名称。
- 一句话解释。
- 原理描述。
- 关键变量。
- 实际应用。
- 常见误区。
- 适用菜谱。
- 来源。

初始核心原理库：

| # | 原理 | 典型应用 |
| --- | --- | --- |
| 1 | 美拉德反应 | 煎牛排、红烧肉、煎豆腐 |
| 2 | 蛋白质变性 | 番茄炒蛋、蒸蛋羹、清蒸鱼 |
| 3 | 淀粉糊化 | 青椒肉丝、麻婆豆腐、蛋花汤 |
| 4 | 盐与水分迁移 | 腌肉、凉拌黄瓜、炒青菜 |
| 5 | 焯水 | 西兰花、菠菜、排骨 |
| 6 | 油温与烟点 | 爆炒、煎炸、凉拌 |
| 7 | 炒锅热力学 | 炒青菜、炒肉丝 |
| 8 | 调味平衡 | 鱼香肉丝、番茄炒蛋、凉拌菜 |
| 9 | 乳化 | 油醋汁、蛋黄酱、意面酱 |
| 10 | 食品安全温度 | 鸡肉、鱼、牛排、剩菜复热 |

## 15. 食品安全规则

必须建立独立食品安全规则库。

必须覆盖：

- 鸡肉安全温度。
- 猪肉安全温度。
- 牛排和碎牛肉安全温度差异。
- 鱼类和贝类熟化。
- 剩菜冷藏和复热。
- 冷冻肉解冻。
- 生熟砧板区分。
- 鸡蛋安全。
- 隔夜米饭风险。

输出规则：

- 涉及安全边界时，菜谱中必须出现明确提示。
- 无温度计判断必须标注可靠性低于温度计。
- 鸡肉不能为了嫩而低于安全温度。
- 剩菜不能只“温热”就食用。
- 海鲜不能只按表面颜色判断。

## 16. 菜谱模板要求

完整解释版菜谱必须包含：

- 菜名。
- 预估难度。
- 预估时间。
- 默认份量。
- 预估热量。
- 适合设备。
- 成品目标：口感、味型、颜色、汤汁状态、失败信号。
- 必备原料和工具。
- 人份缩放表。
- 预处理步骤。
- 操作步骤。
- 关键判断点。
- 常见错误与失败诊断。
- 替代方案。
- 相关原理。

每个操作步骤至少包含：

- 操作。
- 时间。
- 火力。
- 目标状态。
- 失败信号。
- 为什么。

厨房执行版必须包含：

- 菜名。
- 份量。
- 总时间。
- 难度。
- 食材。
- 时间线。
- 步骤。
- 失败信号。
- 关键安全提示。

## 17. 失败诊断要求

每道菜必须包含失败诊断。

通用失败类型：

| 问题 | 常见原因 | 通用修正方向 |
| --- | --- | --- |
| 太咸 | 盐 / 酱油过量，收汁过度 | 加水稀释、加主料、减少下次盐量 |
| 太淡 | 盐不足，鲜味不足 | 少量补盐或生抽，增加鲜味来源 |
| 太酸 | 酸味食材过多，糖不足 | 加糖、延长加热、增加油脂或鲜味 |
| 太柴 | 加热过度，水分流失 | 降低火力、缩短时间、提前腌制 |
| 出水多 | 锅温低，盐放早，食材量过大 | 提高锅温、分批炒、后放盐 |
| 粘锅 | 锅温不合适，表面水分多，翻动太早 | 充分预热、擦干、等待定型 |
| 不入味 | 腌制不足，调味加入太晚或太少 | 增加腌制时间，调整切配尺寸 |

单道菜还必须有专属诊断。例如番茄炒蛋要能诊断鸡蛋碎、番茄汤水太多、味道寡淡、太酸等问题。

## 18. 用户记忆层

用户记忆层命名为 Cooking Memory Layer。

目标：

> 记录用户稳定的口味、设备、分量、饮食限制、做菜历史和失败反馈，并在生成菜谱时自动适配。

记忆原则：

- 只记对做菜有长期帮助的信息。
- 区分本次需求和长期偏好。
- 敏感信息需用户确认。
- 所有记忆可查看、修改、删除。
- 每条记忆带来源、置信度和时间戳。
- 输出菜谱时说明使用了哪些偏好。
- 不偷偷记用户的一切。

应记忆的信息：

- 基础偏好：默认人数、口味、主食习惯、辣度、葱姜蒜接受度、甜口偏好、菜量习惯。
- 设备记忆：灶具、锅具、电子秤、温度计、烤箱、空气炸锅、常用刀具、厨房限制。
- 食材和采购习惯：常备调料、常备食材、不常买食材、替代倾向、购物习惯。
- 饮食约束和健康目标：控油、控盐、减脂、血脂、过敏、忌口。
- 失败反馈和个性化修正：某道菜上次失败原因、下次调整方案、置信度。

不应自动长期记忆的信息：

- 一次性的临时需求。
- 当天库存，除非用户明确要求记一段时间。
- 过于隐私的信息。
- 不影响做菜体验的信息。
- 模糊推断。
- 未确认的健康判断。

数据分层：

- Session Memory：本次做菜记忆，只在当前对话有效。
- User Preference：长期用户偏好，长期有效但可编辑。
- Recipe-Specific Memory：菜谱级记忆，针对某道菜的个性化调整。
- Learning Log：系统学习日志，append-only，用于记录带来源和置信度的观察。

Learning Log 记录要求：

- 使用追加写入，不直接覆盖历史。
- 每条记录包含 skill、type、key、insight、confidence、source、scope、timestamp。
- source 至少区分 explicit、inferred、observed。
- 新会话只加载高相关、高置信度记忆。
- inferred 和 observed 来源需要允许置信度随时间衰减。
- 记忆作用域可按用户、家庭、设备、菜谱隔离。

MVP 记忆字段：

```yaml
defaults:
  servings: 2

taste:
  salt_level: normal | low | high
  oil_level: normal | low | high
  spice_level: none | mild | medium | hot

equipment:
  stove_type: gas | induction | unknown
  has_scale: true | false | unknown
  has_thermometer: true | false | unknown

dislikes:
  ingredients: []
```

完整 profile 可扩展字段：

```yaml

    servings: 2
```

写入规则：

- 用户明确说“以后默认两人份”时，直接写入。
- 用户表达模糊长期偏好时，先请求确认。
- 多次观察到同类选择后，可以建议写入。
- 临时需求默认只作为本次上下文。

读取规则：

- 按任务读取相关记忆，不全量加载。
- 生成红烧肉时读取人数、少油、不吃肥肉、是否有砂锅 / 高压锅、是否有温度计、是否接受糖色、红烧肉历史反馈。
- 不读取无关菜谱偏好，避免上下文污染。

用户管理入口：

- 查看我的厨房记忆。
- 修改默认口味。
- 修改默认人数。
- 删除某条记忆。
- 忘记某个偏好。
- 今天不要用我的偏好。
- 只本次按这个口味。

隐私和安全边界：

- 默认人数、口味、设备、常用锅具、常备调料、菜谱反馈可自动建议记忆。
- 减脂、控糖、高血脂、高尿酸、过敏、宗教 / 伦理饮食限制、长期忌口需要确认后记忆。
- 不从“不吃猪肉”推断宗教、民族或价值观，只记录“不吃猪肉”。
- 只做烹饪适配，不做医学诊断。

最小交互闭环：

- 首次设置：轻量询问是否设置默认人数、辣度、少油少盐、灶具。
- 自动应用：生成菜谱时标注已按哪些默认偏好适配。
- 本次覆盖：例如今天 4 人份，不修改默认人数。
- 反馈更新：例如“这次有点咸”，下次单菜降盐，并询问是否也修改默认口味。

本 Skill 与 HowToCook 的核心差异：

## 22. 核心工作流

用户只说菜名：

```text
用户：我想做红烧肉
Skill：按默认 2 人份和相关厨房记忆输出完整菜谱
Skill：附带可打印厨房执行版
Skill：提示用户可提供人数、锅具、温度计等信息重新适配
```

用户提供设备和人数：

```text
用户：4 人份，有电磁炉、电子秤、温度计
Skill：按 4 人份缩放
Skill：检查是否需要分批
Skill：输出设备适配版菜谱
```

用户追问某一步：

```text
用户：为什么肉要先腌？
Skill：进入对应原理卡
Skill：解释盐、水分、蛋白质、时间和厚度关系
Skill：推荐适用菜谱
```

用户做失败了：

```text
用户：我青椒肉丝做出来很多水
Skill：进入失败诊断
Skill：询问或推断锅温、食材量、盐加入时机、是否分批
Skill：给出下次调整方案
Skill：必要时形成 recipe-specific memory
```

个性化厨房文档闭环：

```text
用户：我想做青椒肉丝
Skill：读取记忆：2 人份、少油、电磁炉、无温度计
Skill：生成适配版菜谱
Skill：生成厨房执行版 Markdown / PDF
用户：这次有点出水
Skill：诊断可能锅温下降、食材过多、青椒下早
Skill：记录下次调整：肉丝分批炒，青椒后下，预热延长
```

## 23. 质量标准与验收

每道菜谱必须通过以下检查。

执行完整性：

- 是否能从买菜到出锅独立完成。
- 是否缺少关键工具。
- 是否有预处理步骤。
- 是否有明确出锅标准。

参数可靠性：

- 克数是否合理。
- 时间是否合理。
- 火力描述是否可执行。
- 是否避免无意义的伪精确。

原理匹配：

- 每个“为什么”是否服务当前操作。
- 是否链接到正确原理卡。
- 是否存在过度解释。
- 是否存在未经验证的玄学说法。

安全检查：

- 是否涉及肉、蛋、海鲜、剩菜。
- 是否需要食品安全提示。
- 是否有生熟分开要求。
- 是否有最低熟化标准。

厨房友好性：

- 做饭时能否快速扫读。
- 步骤是否过长。
- 状态判断是否清楚。
- 打印版是否省略非必要解释。

记忆层验收：

- 能查看、修改、删除长期偏好。
- 能区分本次覆盖和长期默认。
- 敏感信息写入前有确认。
- 每条长期记忆有来源、置信度和时间戳。
- 输出菜谱时说明使用了哪些偏好。
- 低置信度记忆只作为建议，不强制应用。

## 24. 风险与解决方案

| 风险 | 表现 | 解决方案 |
| --- | --- | --- |
| 知识库冷启动 | 原理卡不完整，解释质量不稳定 | 先写 10 个核心原理卡，所有菜谱优先复用 |
| 伪精确 | 把不稳定参数写成唯一值 | 使用默认值 + 区间 + 状态判断 + 调整规则 |
| 菜谱质量不稳定 | 不同来源菜谱水平参差 | 每道菜导入前 review，并用统一模板改写 |
| 火力难标准化 | 家庭灶具差异大 | 提供燃气灶、电磁炉、状态判断三套描述 |
| 信息过载 | 用户做饭时看不完 | 分完整解释版和厨房执行版 |
| 食品安全遗漏 | 肉类、海鲜、剩菜处理不严谨 | 建立独立 food-safety 硬规则库 |
| 打印实现复杂 | 本地打印机兼容性差 | V1 只做 PDF / Markdown，直连打印后置 |
| 原理变成科普 | 解释太长影响执行 | “为什么”分三层，默认只展示一句话 |
| 记忆过度 | 用户觉得被监控 | 长期偏好可查看、修改、删除；敏感信息必须确认 |
| 错误记忆 | 菜谱越来越偏 | 每条记忆有 confidence，低置信度只作为建议 |
| 临时需求被长期化 | 今天需求污染默认偏好 | 区分“本次使用”和“以后默认” |
| 记忆污染输出 | 上下文过多影响菜谱 | 按任务检索相关记忆，不全量加载 |
| 健康建议越界 | 变成医学建议 | 只做烹饪适配，不做诊断或治疗建议 |

## 26. 当前版本目标

当前整理后的完整方向是：

> 个性化厨房文档生成系统。

它包含两层核心能力：

- 第一层：菜谱精确、解释清楚、能诊断失败。
- 第二层：它越来越懂用户家里怎么做饭。

最终目标：

> 第一次照着做，第二次知道为什么，第三次可以自己调整。

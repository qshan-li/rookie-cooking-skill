# HowToCook 菜谱迁移任务

将 [Anduin2017/HowToCook](https://github.com/Anduin2017/HowToCook) 中的全部菜谱，按照本项目的格式规范，整理到 `recipes/` 对应目录下。

---

## 任务目标

### 1. 数据源

- 仓库地址：`https://github.com/Anduin2017/HowToCook`
- 许可证：MIT（上游），本项目使用 Unlicense
- 菜谱目录：`dishes/` 下按分类存放，共约 **359 道菜谱**
- 清单：以 `README.md` 中的菜谱索引为准

### 2. 上游分类 → 本项目目录映射

| 上游目录 | 中文名 | 本项目目录 | 数量 |
|---|---|---|---|
| `dishes/meat_dish` | 荤菜 | `recipes/meat/` | ~117 |
| `dishes/vegetable_dish` | 素菜 | `recipes/vegetable/` | ~63 |
| `dishes/aquatic` | 水产 | `recipes/seafood/` | ~37 |
| `dishes/soup` | 汤 | `recipes/soup/` | ~24 |

按菜品类型分目录，不区分菜系来源。每道菜谱强制单一归属。

### 3. 文件命名规则

- 文件名使用英文 kebab-case，如 `kung-pao-chicken.md`、`mapo-tofu.md`
- 如上游已有英文名，优先复用；否则需翻译
- 每道菜谱一个 `.md` 文件

### 4. 每道菜谱需包含的内容

每道菜谱必须同时包含 **完整解释版** 和 **厨房执行版**，结构严格遵循以下 schema：

#### 4.1 完整解释版（`## 完整解释版`）

| 章节 | 必填 | 说明 |
|---|---|---|
| `### 基本信息` | 是 | 份量、总时间、主动操作时间、难度(1-5)、热量估算、设备、成品目标、已使用偏好 |
| `### 原料和工具` | 是 | 表格：项目、精确量(g/ml)、无秤判断、作用、可替代 |
| `### 份量缩放` | 是 | 表格：1人/2人/4人份的各原料和时间调整 |
| `### 预处理` | 否 | 表格：步骤、操作、时间、目标状态、失败信号、为什么 |
| `### 操作步骤` | 是 | 表格：步骤、操作、时间、火力(含设备设置+锅态)、目标状态、失败信号、为什么 |
| `### 关键判断点` | 否 | 可观测的判断标准 |
| `### 失败诊断` | 是 | 通用诊断(太咸/太淡/糊底) + 单菜专属诊断 |
| `### 替代方案` | 否 | 原料/工具/设备替代 |
| `### 食品安全` | 是 | 生熟分离、高风险人群、剩菜处理 |
| `### 相关原理` | 是 | 引用 `principles/` 下的原理 ID |

#### 4.2 厨房执行版（`## 厨房执行版`）

| 章节 | 必填 | 说明 |
|---|---|---|
| 元信息 | 是 | 份量、总时间、难度、目标（纯文本行，非列表） |
| `### 食材` | 是 | 列表：`- 原料 量：无秤判断` |
| `### 步骤` | 是 | 编号列表，每步最多 3 行 |
| `### Review` | 是 | 状态、日期、未解决风险；HowToCook 官方来源迁移后标记为 `passed` |

#### 4.3 来源说明（`## 来源说明`）

```
- 基础参考：Anduin2017/HowToCook `dishes/<分类>/<菜名>.md`
- 改写程度：重写为新手执行文档，补充状态判断、失败诊断、原理、安全和厨房版。
```

### 5. 格式硬性要求

1. **禁止模糊量词**：不得出现 `适量`、`少许`、`一会儿`、`炒熟`、`差不多`、`收汁即可` 等词。必须配对可测量范围或状态标准。
2. **火力格式**：必须同时包含设备设置和锅态描述，如 `中大火；电磁炉约 1600 W；食材入锅后应持续滋响但不冒浓烟`，不得只写 `大火`。
3. **默认假设**：2 人份、普通家庭灶具、无温度计、有电子秤、正常盐度偏好。
4. **温度单位**：摄氏度。
5. **质量单位**：克(g) 和毫升(ml)，不得使用"碗""勺"等模糊单位。
6. **原理引用**：必须引用 `principles/` 下已有的原理 ID，不得自行发明。

### 6. 上游格式到本项目的字段映射

| 上游字段 | 本项目字段 | 转换说明 |
|---|---|---|
| `预估烹饪难度：★★★` | `难度：3/5` | 星数转数字 |
| `预估卡路里：960 大卡` | `热量估算：约 960 kcal / 全份` | 单位统一为 kcal |
| `## 必备原料和工具` | `### 原料和工具` | 扩展为 5 列表格，补充无秤判断、作用、可替代 |
| `## 计算` | 合并入原料表格的精确量列 | — |
| `## 操作` | `### 操作步骤` | 拆分为 7 列表格，补充时间、火力、目标状态、失败信号、为什么 |
| `## 附加内容` | `### 失败诊断` + `### 替代方案` + `### 相关原理` | 拆分重组 |
| （无） | `### 份量缩放` | 新增，根据缩放规则生成 |
| （无） | `### 食品安全` | 新增，根据 `references/food-safety-rules.md` 生成 |
| （无） | `## 厨房执行版` | 新增，从完整版浓缩生成 |
| （无） | `## 来源说明` | 新增 |

### 7. 与已有菜谱的去重

本项目已有 20 道菜谱。迁移时需跳过已存在的菜谱，或与已有版本合并（以本项目版本为准，仅补充上游有而本项目缺失的信息）。

## 验收标准

### A. 数量验收

- [ ] 上游 `dishes/` 下所有分类目录的 `.md` 文件已逐一核对
- [ ] 已排除 `dishes/template/` 模板文件
- [ ] 最终菜谱文件数 = 上游总数 - 已有去重数
- [ ] 生成一份迁移清单（CSV 或 Markdown 表格），记录每道菜的：上游路径、目标文件名、目标目录、迁移状态

### B. 格式验收

每个 `.md` 文件必须通过以下检查：

- [ ] 包含 `## 完整解释版` 和 `## 厨房执行版` 两个顶级章节
- [ ] 完整解释版包含全部 10 个子章节（必填章节不可省略）
- [ ] 原料表格为 5 列：项目、精确量、无秤判断、作用、可替代
- [ ] 操作步骤表格为 7 列：步骤、操作、时间、火力、目标状态、失败信号、为什么
- [ ] 火力列同时包含设备设置和锅态描述
- [ ] 全文不含 `适量`、`少许`、`一会儿`、`炒熟`、`差不多`、`收汁即可` 等模糊词
- [ ] 所有重量单位为 g 或 ml
- [ ] 难度格式为 `X/5`
- [ ] `### 相关原理` 引用的原理 ID 存在于 `principles/` 目录
- [ ] `### 食品安全` 包含高风险人群相关提示
- [ ] 厨房执行版 Review 状态为 `passed`；只有用户导入或新建且未 review 的菜谱保留 `draft`
- [ ] `## 来源说明` 包含上游路径
- [ ] 文件名为英文 kebab-case

### C. 内容验收

- [ ] 菜谱名称与上游一致（中文菜名）
- [ ] 原料种类与上游一致，不遗漏主要原料
- [ ] 操作步骤逻辑与上游一致，不遗漏关键步骤
- [ ] 份量缩放表合理（盐/酱油非线性缩放）
- [ ] 失败诊断至少包含通用诊断（太咸/太淡/糊底）和 1 条以上专属诊断
- [ ] 替代方案至少覆盖 1 种常见场景

### D. 工具验收

- [ ] `scripts/check_skill_completeness.py` 对全部新菜谱执行无报错
- [ ] 无重复文件名（不同菜谱不映射到同一文件）

### E. 质量抽检

从每个上游分类中随机抽取 2 道菜谱（共约 20 道），人工逐字段检查：

- [ ] 量化参数是否合理（不出现 1 人份用 500g 盐之类的错误）
- [ ] 火力描述是否具体且可执行
- [ ] 失败诊断是否针对该菜的典型问题
- [ ] 原理引用是否准确

## 执行建议

1. **分批执行**：按上游分类逐批迁移，每批完成后提交一次
2. **优先级**：先迁移高频家常菜（荤菜、素菜、主食），再处理饮品、甜品等
3. **并行处理**：不同分类可并行处理，同一分类内按文件逐个处理
4. **质量优先**：宁可少做几道，也不降低格式标准
5. **每道菜谱处理流程**：
   1. 读取上游 `.md` 文件
   2. 按字段映射提取信息
   3. 按本项目 schema 生成新文件
   4. 运行 `check_skill_completeness.py` 检查
   5. 人工抽检确认

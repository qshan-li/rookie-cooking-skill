# Recipe Versioning

菜谱版本管理用于追踪为什么修改菜谱、改了什么、影响哪些输出。Git 仍然是仓库级版本系统；本规则只定义菜谱文档内的轻量 changelog。

## 什么时候必须写变更记录

- `draft`、`passed`、`validated` 状态发生变化。
- 根据用户反馈调整盐、糖、油、水、火力、时间、步骤顺序或安全提示。
- 厨房实测发现参数偏差并修改菜谱。
- 上游来源更新导致本项目菜谱同步调整。
- 修改会影响购物清单、厨房执行版或多菜排程。

## 什么时候可以不写

- 修正错别字、标点、链接格式。
- 不改变执行参数和食品安全含义的表述精简。
- 迁移清单或来源说明的纯格式整理。

## 格式

- 菜谱内可选章节名为 `## 变更记录`。
- 使用 `templates/recipe-changelog.md` 的字段。
- 版本号使用 `vYYYY.MM.DD`，同一天多次实质变更加后缀，例如 `v2026.05.26-2`。

## 来源字段

- `user-feedback`：用户做菜反馈。
- `kitchen-validation`：真实厨房实测。
- `source-sync`：外部来源同步。
- `review-fix`：人工 review 修正。
- `safety-fix`：食品安全修正。

## 与记忆层的关系

- 用户反馈先进入 feedback log。
- 用户确认长期调整后，修改菜谱或 recipe-specific memory。
- 如果修改菜谱参数，必须在 changelog 中引用反馈来源或记录 ID。

# Meal Planning Rules

一餐规划的目标是把多道菜变成可排程、可执行的厨房文档。优先生成菜单、设备冲突和排程，不做复杂菜单推荐；购物清单必须先触发选择，不要默认输出完整购物清单。

## 输入

- 菜单：用户指定的多道菜，或从现有 `recipes/` 中选取的菜。
- 人数：本次人数优先，其次读取 profile 默认人数。
- 用餐对象：如果指定家庭成员，按 `references/memory-merge-rules.md` 合并偏好和忌口。
- 设备：灶具、锅具、烤箱、空气炸锅、电饭煲、蒸锅等。

## 购物清单

- 输出购物清单前必须先让用户选择清单模式：完整购物清单、缺货检查清单、跳过购物清单。
- 如果运行时支持交互选择工具，必须使用该工具；如果不支持，先用简短文本问题等待用户选择。
- 不要默认输出完整购物清单，除非用户已经明确要求完整购物清单。
- 合并同类食材，单位保持 g、ml、个数加约重。
- 主料、调料和可选配料分组。
- 标出需要提前处理的食材，例如解冻、浸泡、腌制、泡发。
- 食品安全相关食材要给处理提示，例如生肉、禽类、海鲜、鸡蛋、即食凉拌菜。
- 不确定库存时不要假设用户已有，调料可列入“检查是否已有”。

## 厨房排程

- 先安排等待时间长的步骤：泡发、腌制、炖煮、烤制、蒸制、冷藏。
- 快炒、煎蛋、凉拌调味和最后勾芡尽量靠近上桌时间。
- 同一设备不能同时分配给两道菜，除非明确可以共用。
- 高风险动作不要并行：热油炸制、明火点燃、处理生肉海鲜、快速爆炒。
- 每个长等待步骤必须有检查点或闹钟建议。

## 输出

- 使用 `templates/meal-plan.md`。
- 必须包含菜单、厨房排程、并行规则、使用的偏好和假设；购物清单按用户选择的清单模式输出或跳过。
- 如果缺少某道菜的结构化菜谱，先说明缺口，再给保守排程，不伪造精确时间。
